Recommended Git Workflow Git Workflow Recomendado

Feature Branch + Pull Rebase + Squash Merge Feature Branch + Pull Rebase + Squash Merge

Phase 1: Feature Preparation Fase 1: Preparación de la Feature

1. Make sure you are up to date on dev:1. Asegurarte de estar actualizado en dev:

git checkout dev
git pull origin dev

2. Create a new feature branch:2. Crear nueva feature branch:

git checkout -b feature/user-authentication

3. Work on the feature (multiple commits):3. Trabajar en la feature (múltiples commits):

# ... make changes ...
git add .
git commit -m "feat(auth): add login form validation"
# ... more changes ...
git commit -m "fix(auth): handle edge case in password validation"
# ... more changes ...
git commit -m "feat(auth): implement session management"

Phase 2: Sync with Dev (Before Merge) Fase 2: Sincronización con Dev (Antes del Merge)

4. Update local dev:4. Actualizar dev localmente:

git checkout dev
git pull origin dev

5. Go to the feature branch and rebase on top of the updated dev:5. Ir a feature y hacer rebase sobre dev actualizado:

git checkout feature/user-authentication
git rebase dev

6. Resolve conflicts if they exist during the rebase:6. Resolver conflictos si existen durante el rebase:

# (edit files, resolve conflicts)
git add resolved_file.java
git rebase --continue
# Repeat until the rebase is complete

Phase 3: Integration Options Fase 3: Opciones de Integración

Option A: Squash Merge (Recommended for a Clean History) Opción A: Squash Merge (Recomendado para Historia Limpia)

7. Switch back to dev:7. Volver a dev:

git checkout dev

8. Squash merge (combines all feature commits into one):8. Squash merge (combina todos los commits de feature en uno):

git merge --squash feature/user-authentication

9. Create a descriptive commit message:9. Crear commit descriptivo:

git commit -m "feat(auth): implement complete user authentication system

- Add responsive login form with client-side validation
- Implement secure password hashing with bcrypt
- Add JWT-based session management
- Include login/logout functionality
- Add authentication middleware

Closes #123"

10. Push to remote:10. Push a remoto:

git push origin dev

11. Delete the feature branch (no longer needed):11. Eliminar rama feature (ya no necesaria):

git branch -d feature/user-authentication
git push origin --delete feature/user-authentication

Option B: Regular Merge (Preserves Detailed History) Opción B: Merge Regular (Preserva Historia Detallada)

As an alternative to squash merge, a regular merge will preserve all the individual commits from the feature branch.Como alternativa al squash merge, un merge regular preservará todos los commits individuales de la rama de feature.

# Alternative to squash merge
git checkout dev
git merge feature/user-authentication  # May result in a merge commit
git push origin dev

Advantages of This Workflow Ventajas de Este Workflow

AspectAspecto BenefitBeneficio
git pull origin dev Keeps your local branch up-to-date with the team's changes.Mantiene tu rama local actualizada con los cambios del equipo.
git rebase dev Applies your changes on top of the latest changes from dev, avoiding unnecessary merge commits.Aplica tus cambios encima de los últimos cambios de dev, evitando merge commits innecesarios.
git merge --squash Turns all "noisy" WIP commits into a single clean, atomic commit.Convierte todos los commits "ruidosos" en un commit limpio y atómico.
Linear HistoryHistoria lineal Facilitates git bisect, code review, and debugging.Facilita git bisect, code review y debugging.
Atomic CommitsCommits atómicos Each feature is a complete logical unit that can be easily reverted.Cada feature es una unidad lógica completa que se puede revertir fácilmente.

Important Rules Reglas Importantes

DO:

  • Always rebase before merging for a clean history.Siempre rebase antes del merge para historia limpia.
  • Use squash merge for features with multiple "work-in-progress" commits.Usar squash merge para features con múltiples commits "trabajo en progreso".
  • Pull with rebase on feature branches: git pull --rebase origin dev.Pull con rebase en feature branches: git pull --rebase origin dev.
  • Resolve conflicts during the rebase, not after.Resolver conflictos durante el rebase, no después.

DON'T:

  • Never rebase shared branches (like dev or master).Nunca hagas rebase de ramas compartidas (como dev o master).
  • Do not rebase after creating a Pull Request if others are reviewing it.No hagas rebase después de crear Pull Request si otros están revisando.
  • Do not force push to shared branches.No hagas push force a ramas compartidas.
